# 03 ─ 審查（一）：垃圾偵測技術

> 「最好的方式就是用它的魔法打敗它——它在發 spam，我們當然就用語言模型來偵測它到底在講什麼。」〔TWIGF〕

本章拆解 Matters 怎麼把「垃圾」從每天 2–3 萬篇新內容裡找出來。核心約束始終是那句不對稱原則：**寧可漏看一篇垃圾，不願消音一篇表達。**〔提案〕〔spam:D1〕

## 3.1 兩條防線：行為層與內容層

從一開始，反濫用就分兩路打：〔過濾〕

- **防止帳號機器人**：從「行為」判斷，在源頭減緩無效操作（大量註冊、刷讚）。
- **防止內容機器人**：從「內容」辨識與過濾，維持公共頁面的閱讀體驗。

這兩層對應到不同的技術，也對應到不同的演化時程。

## 3.2 行為層：reCAPTCHA → Turnstile（2020.04）

2020 年 4 月先導入 Google reCAPTCHA，放在**註冊、登入、文章讚賞**三處，從源頭擋掉機器人。但 reCAPTCHA 需手動驗證、打斷操作、維護成本高，後期改採 **Cloudflare Turnstile**，多數情況可在背景完成辨識，體驗更好。〔過濾〕

值得注意的是行為層的「退場」：2025 年 3 月，因內容層的 spam 模型已足夠有效，團隊**停用了 reCAPTCHA 與 Email domain 名單限制**。〔時間線〕這是「模型成熟後，舊防線可以拆」的一個具體例子。

## 3.3 內容層的誕生（2024.08）

隨著違規創作數量龐大、內容農場濫用 SEO、機器人發文不斷進化，**傳統關鍵字過濾無法判斷上下文**。恰逢語言模型成熟，團隊改用語言模型建立內容分析模型。〔過濾〕

**為什麼選 Encoder（BERT 類）而非 Decoder（GPT 類）**：任務是「理解語意、判斷字詞關係」，不是生成；Encoder 模型小、推理快、硬體門檻低，適合 Matters 規模。〔過濾〕白話版：

> 「一種就是創作型的、預測性的；另一種是推理、去理解你在說什麼——我們要的是後者。」〔TWIGF〕

**模型選型與訓練**：

- 樣本：運營在日常值班中人工標記違規／正常文章，初期累積約 **4 萬多篇**正負樣本。〔過濾〕
- 選型：把樣本餵給不同廠牌模型評估，與人工辨識結果越接近者越適合，最終選 `ibm-granite/granite-embedding-107m-multilingual`，搭配 **LoRA** 微調。〔過濾〕〔排序報告〕
- 部署：每篇文章發布後內容被清成純文字，送 AWS Lambda 推論，得到一個分數。〔過濾〕

**處置原則（關鍵）**：被判違規的文章**不會被刪除，只是不出現在公共頁面**（首頁精選、熱文、各頻道列表）。〔過濾〕這條原則往後貫穿全平台（第 06 章）。

**成效**：

| 指標 | 上線初 | 後續 |
|---|---|---|
| 攔阻無效文章成功率 | 96% | **99.7%** |
| 模型誤判使用者文章比例 | 3.1% | **0.13%** |

〔過濾〕

## 3.4 模型怎麼判：信任分數與三段式決策

模型對每篇內容吐出一個分數——也就是簡報所稱的「信任分數（trust score）」。白話：「越高分越可能是 spam，它是把每一段、每一句拆解分析再加總出來的。」〔TWIGF〕

決策政策（程式碼 `policy.py`）是三段式：〔排序報告〕

| 分數帶 | 動作 |
|---|---|
| < 0.35 | 直接放行（ALLOW） |
| 0.35–0.85 灰區，或帶高風險訊號 | 丟 LLM 覆核（REVIEW） |
| ≥ 0.85 且命中高風險訊號 | 封鎖（BLOCK） |

**高風險訊號**：外部連結、聯絡方式（line／telegram／微信／賴…）、被混淆的聯絡方式、成人服務關鍵字（私約、外送茶、包夜…）。〔排序報告〕

> **【內部／敏感參數】** 訓練細節：token 上限 512；LoRA r=8、α=32；學習率 2e-5；batch 128；以 `roc_auc` 選模；門檻掃描以 `bias = 2·FP + FN`（刻意偏重低誤殺）。部署於 AWS Lambda 容器映像 + API Gateway。〔排序報告〕——這些是可被惡意者用來規避的參數，依分層揭露原則，網站公開版本延後或抽象化處理。

## 3.5 留言模型獨立化

留言和長文是兩種對手：留言短、訊號少、歧義高、且高度模板化（約 50 個模板家族）。因此團隊把留言獨立成一個模型：**multilingual-e5-small**（約 0.1B 參數）+ 邏輯回歸頭。〔演算法筆記〕〔spam:D1〕

- **偽陽性僅 0.33%**；leave-one-family-out 召回 **0.88**，遠優於舊共用模型的 **0.68**。〔演算法筆記〕〔spam:D1〕
- **反直覺的發現——小模型勝過笨重模型**：「granite 是 IBM 做的，規模跟 e5 差不多，但我們用 LoRA 微調……在留言這種極短字串中，錯誤率比 e5 高很多。」〔演算法筆記〕
- 訓練資料直接來自守望相助隊累積的真實移除紀錄（正樣本）與正常留言（負樣本），形成「群眾標註 → 模型」的閉環（見第 08 章）。〔repo〕

## 3.6 兩代模型並存

線上仍保有一個較古老的開源 BERT 分類器；新的 `spam-detection-scaffold` 鷹架則是 granite embedding + LoRA + LLM 覆核層。兩者並存、分工服務不同場景。〔排序報告〕〔repo〕

## 3.7 人工標籤：模型的真相從哪來

模型的「正解」全部來自人。標籤來源被分級記錄並賦不同權重，**人工確認的標籤權重最高**（訓練權重 2.0／1.0／0.5），用以抑制 over-kill。〔repo〕〔spam:D2〕

資料持續回流再訓練：違規文章一旦出現在公共頁面就被標記，作為下一輪樣本；使用者也可回報自己的文章未出現於公共頁面，由團隊確認後標註。經過幾輪再訓練，誤判率持續下降（即 3.3 的 3.1%→0.13%）。〔過濾〕

## 3.8 模型會「自我中毒」：錯殺放大

這是全份材料中最重要的技術治理洞見之一：**舊模型在每天搜集資料重訓的過程中，會逐漸被自己污染，誤判率漂移——而且不是被惡意協同攻擊導致，而是被「錯殺放大」導致。**〔演算法筆記〕

- **機制**：一個帳號某天的文章被標記有問題，後來變正常，但那些正常文仍被識別為有問題。「就像不小心把可樂倒進麵團裡，越揉越分不出可樂在哪裡。」結果政治評論、學術考據、日記、詩，都被看成垃圾文。〔演算法筆記〕
- **安全網救了它**：「好在基於相對安全的設計，這些分數僅見於後台，並沒有被真正落實在演算法的排序上面。」〔演算法筆記〕——這正是「分數歸分數、排序歸排序」分層設計的價值（第 06–07 章詳述）。
- **修正方法**：用 Claude Sub-agent（Sonnet 模型）重新校正 **5,000 筆**資料，成本約 **1 USD**，**偽陽性降低 10 倍以上**，進入可上線階段。〔演算法筆記〕
- **意外收穫**【政治敏感】：校正時抓出「許多防詐騙科普，其實是貸款公司的洗白文」——正是這類文章，當年觸動了 DNS-RPZ，導致整個 Matters 網域在台灣被封（見第 10 章）。〔演算法筆記〕

由此導出一個成本／成效結論：

> 「『小模型（0.1B）+ LoRA 微調』+『LLM（可能超大幾 T 參數）校正垃圾文品質』的混搭模式，可能比更新模型的效果還要好。」〔演算法筆記〕

## 3.9 軍備競賽：模型不動，對方會動

偵測的根本困難在於對抗性：

> 「模型訓練好之後它是不會動的，但是對方會動。」〔TWIGF〕

- **垃圾的進化（運營白話）**：一開始把關鍵字打在標題，被抓後改成「標題、內文看起來很正常，然後在下面下面第四頁的某一段偷偷藏一個關鍵字」，「有點像是你追我躲」。〔TWIGF〕
- **AI 生成變異後更難抓**：「道高一尺，魔高一丈。」〔演算法筆記〕
- **對策——回訓迴圈**：「定期回收之前沒被判斷出來的 spam，再重新訓練一次。」〔TWIGF〕
- **哲學**：「這種演算法其實就是一種魔法，魔法什麼人都可以用，有好的魔法也有壞的魔法……如何使用機器對機器？」〔TWIGF〕

## 小結

偵測技術走的是一條「行為層擋機器人 → 內容層理解語意 → 留言獨立小模型 → LLM 校正品質」的路線，每一步都壓低誤判、同時維持公共頁面的可讀性。但有一種對手，單看一篇分數很低、卻是一整群帳號用同一套模板協同洗版——它必須靠「行為而非語義」才會現形。下一章談集團（Ring）。
