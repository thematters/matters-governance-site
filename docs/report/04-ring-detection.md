# 04 ─ 審查（二）：集團掃描（Ring）與短文／動態的後門

## 4.1 為什麼單篇分數不夠

第 03 章的模型是對「一篇內容」打分。但有一種對手，單看一篇分數很低，卻是一整群帳號用同一套模板協同洗版。最具代表性的例子：某「高雄翻譯社」廣告，單則模型分數只有 **0.09**，幾乎被當成正常內容——但它在「集團」視角下，整群現形。〔repo〕〔spam:D4〕

這就是 **Ring（集團）偵測**要解決的問題：**抓的是行為，不是字句。**

## 4.2 Ring 的本質

Ring 的訊號是**行為性的，不是語義性的**：同一個「模板指紋」被大量新帳號、低信用或拋棄式帳號在短時間內重複張貼。〔repo〕用主持人的白話說：

> 偵測「重複樣態的內容在短時間內出現」，幾乎可以確定是 Ring，可以一網打盡。〔演算法筆記〕

換句話說，個別帳號可能聰明地讓每篇分數壓在門檻以下，但**「大量帳號＋幾乎相同內容＋短時間」這個行為樣態本身**，就是最強的證據。

## 4.3 怎麼抓：兩階段

Ring 偵測分兩階段，先粗篩、再精算：〔repo〕

**第一階段——資料庫粗篩**（`sql/detect_spam_rings.sql`）
- 正規化：去除 HTML、把網址／@提及／數字遮罩掉。
- 取內容前 200 字的雜湊（md5）作為「模板指紋」。
- 依指紋分組，計算「不同作者數」，並過濾新帳號。
- 【內部】此查詢須在 VPC 內、對正式資料庫的讀取副本執行。

**第二階段——應用層精算**
- **繁簡折疊（OpenCC）**：這一步極關鍵。同一句話繁體一條、簡體一條會被當成兩群而漏抓——實例：「披著……」繁體 0 命中，折疊成簡體「披着……」後命中 **28 個帳號**。〔repo〕
- **字元 n-gram Jaccard 相似度** + union-find 把相似內容聚成同一群（POC 用 4-gram、海巡 bot 用 3-gram）。
- **帳號名稱亂碼分數**等輔助訊號。
- POC（`eval/ring_detect_poc.py`）在 5 個真實叢集上驗證成立，其中一個模板橫跨多達 **28 個帳號**。〔repo〕

## 4.4 兩把鑰匙的安全設計

Ring 一旦誤判，傷害是「整群」的，因此處置設計格外保守：〔repo〕

- **雙訊號才動作**：需至少 2 個獨立訊號同時成立，才考慮自動凍結。
- **凍結是可逆的**：帳號層級的 Ring 凍結用 `user_restriction`／`archiveUser`（可回復），不是刪除。
- **影子先行（shadow-first）**：先在不影響使用者的狀態下觀察，再考慮實際動作。
- **現況**：帳號層級的 Ring 凍結仍處於**候選階段，尚未產品化**；目前主要在留言層運作。〔repo〕

## 4.5 海巡 bot 的 Ring 閘門

自動巡邏的「海巡 bot」內建一道 **ring gate**：當守望相助隊已確認移除過 **3 則以上**高度相似的留言，bot 即把符合該樣態者一併處理，**且覆寫模型分數**（正是用來抓「高雄翻譯社 0.09」這種單篇低分、集團現形的情況）。比對同樣用 OpenCC 繁簡折疊 + 字元 3-gram Jaccard ≥ 0.8。〔repo〕bot 的完整護欄（dry-run、kill-switch、反轉率煞車）見第 08 章。

## 4.6 短文與動態：被當成後門

長文有成熟的偵測，於是濫用者轉向**短文與動態（moment）**作為繞過的後門。`spam-detection-scaffold` 的「文章與動態濫用報告」（D2）正是處理這條戰線的偵測、處置與模型矯正。〔spam:D2〕

短文之所以更難：〔演算法筆記〕

- **訊號量少、歧義度高**：140 字以內，難抓上下文。
- **偽陽性更麻煩**：正常短文很容易被誤判。
- Matters 原有短文（「短動態」）過去**刻意不集成**到公共頻道，只在個人頁可見——好處是違規帳號的廣告短文「其他人看不到，體驗不被破壞」。風險是一旦集成、又用互動做排序，「無腦集成就會把一大堆垃圾灌進頻道，上線即陣亡」。〔演算法筆記〕

## 4.7 一個便宜又有效的解法：「閒聊」頻道的社群設計

面對短文集成的兩難，團隊沒有只靠模型，而是用**社群設計**解決——這是全報告最值得政策圈注意的一招：

- 集成短文的「**閒聊**」頻道採 **opt-in 白名單 + 退火（annealing）**：使用者須主動「加入」；管理員看過沒問題、**或 48 小時內沒被拒絕**，就通過取得資格。〔演算法筆記〕
- **成效**：「開放半個月以來，『閒聊』杜絕了 100% 的垃圾內容，即使整個垃圾短文總量佔據整個站台超過一半的比例。」〔演算法筆記〕

> 「這樣的社群設計機制，比用模型偵測還要便宜有效許多。在規模小的 local 社群，其實挺適合的。」〔演算法筆記〕

這句話呼應了全報告的一個底層判斷：**最便宜有效的治理，往往不是更強的模型，而是更好的社群設計與流程。**（守望相助隊是同一個邏輯，見第 08 章。）

## 小結

Ring 偵測補上了單篇模型的盲區：它抓的是「協同行為」而非「個別字句」，並用繁簡折疊、相似度聚群與雙訊號可逆處置，謹慎地對付集團洗版。而短文／動態的後門，最終被一個 opt-in 頻道設計便宜地擋下。下一章回到那條反覆出現的分數線——當門檻劃下去，誰被誤殺？
