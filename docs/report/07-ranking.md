# 07 ─ 首頁排序：大家口中的「演算法」

> 「平台治理真正的權力中心，不在『刪除內容』，而在『決定誰被看見』。」〔排序報告〕

一般人對「演算法」最具體的想像，就是首頁怎麼排序。但要先分清楚兩件常被混為一談的事：**被排到後面（降權）** 和 **被排除在外（過濾）**，是兩種完全不同的機制。

## 7.1 首頁熱門的排序公式

首頁熱門由 `recommendationService` 計算，核心是一條加權公式：〔排序報告〕

```
score = 0.3·read_score + 0.4·comment_score + 0.3·donation_score
```

| 訊號 | 權重 |
|---|---|
| 閱讀 | 0.3 |
| **留言** | **0.4（最高）** |
| 贊助／打賞 | 0.3 |

幾個關鍵設計：〔排序報告〕

- **進榜門檻**：`readers ≥ 5` **或** `comments ≥ 3`。
- **回看窗口**：5 天。
- **時間衰減**（線性，clip 到 0）：`score_i = max(1 − age/(decayDays·86400), 0)`，其中 `decayDays = 3`（3 天線性衰減到 0）。
- **【內部】大戶抑制正規化（v4）**：`factor = min(((median·ε)/(f+ε))^0.15, 1)`，用來壓抑單一大戶洗榜。（此類抗操弄參數依分層原則，公開版抽象化。）

值得注意的是**留言權重最高（0.4）**——這透露了平台想獎勵的是「引發討論」，而非單純的曝光或金錢，與一般以互動／停留時間最大化為目標的商業平台不同。

## 7.2 降權 ≠ 刪除：垃圾是二元的 in／out

最容易被誤解的一點：**垃圾不是被「排到後面」，而是被「整篇剔除出候選池」。** 〔排序報告〕

- 垃圾的排除是**二元的 in／out**——在排序的 SQL 查詢裡直接以 `WHERE` 過濾掉（`common/utils/knex.ts`），不進入排序競爭。
- 判定門檻 `spamThreshold` 存於 `feature_flag` 資料表 + Redis 快取；**調整門檻不必重新部署**——這也是為什麼門檻是個敏感的營運資產（第 03 章）。
- 對比：一般內容彼此之間才是靠上面的熱門公式競爭可見度（降權）；spam 過濾則是更早一步的「能不能進場」。

這個區分很重要，因為它意味著「我的文章還在、卻沒人看到」其實有**兩種完全不同的原因**：一是被 spam 過濾擋在候選池外（降噪／小黑屋），二是進了池子但排序競爭不過（降權）。前者是治理處分，後者是演算法結果——而兩者對使用者來說，體感都是「消失」。

## 7.3 小黑屋的三層實作

第 06 章的「小黑屋」在排序層的具體實作，其實橫跨三張表：〔排序報告〕

1. **使用者層**：`user_restriction(articleHottest / articleNewest)`——把該作者文章從首頁熱文／最新排除（mutation `putRestrictedUsers`）。
2. **內容層**：`article.is_spam` / `article.spam_score`（mutation `setSpamStatus` 或模型寫入）。
3. **推薦設定層**：`article_recommend_setting.{in_hottest, in_newest, in_search}`（mutation `toggleArticleRecommend`）。

效果是「帳號維持 active、文章用直連 URL 仍可開啟，但消失於首頁與公共廣場」。

## 7.4 新首頁（2025.07）與「首頁被弱化」

2025 年 7 月新首頁上線，排序邏輯與權重重新設計。〔時間線〕一個耐人尋味的使用者端觀察來自資深使用者 Denken：〔TWIGF〕

> 演算法一旦公開「你是可以操弄它的，它的熱門度、點擊機制都是可以知道的，也是可以被濫用的。」結果使用者的心聲是「Matters 把首頁弱化掉了，現在首頁已經不是主要流量來源」——「有點回到類 PTT 時代」。

也就是說，面對「公開演算法就可能被操弄」的兩難，Matters 的一種實際選擇是**降低首頁演算法的權力**，讓追蹤、標籤等非演算法路徑承擔更多分流——這本身就是一種治理立場。

## 7.5 正當程序的缺口

當審查從「刪除」轉移到「降權／降噪」，傳統的救濟鏈就斷了。〔排序報告〕

| | 傳統刪除路徑 | 演算法降權路徑 |
|---|---|---|
| 內容 | 被刪除 | **仍在** |
| 通知 | 有 | **無** |
| 使用者發現 | 會 | **不會** |
| 申訴 | 可 | **無從申訴** |
| 救濟 | 司法可介入 | **無從救濟** |

這對應到歐盟 DSA 第 17 條「理由說明」義務尚未解決的三個前沿問題：〔排序報告〕

1. 降權是否觸發「理由說明」義務？
2. 「不推薦」算不算限制？
3. 演算法調整是否必須被解釋？

主持人用「三語同義」點出同一件事的三種語言：**工程**上是 `user_restriction` 與 `spam_score ≥ threshold` 的 SQL 過濾；**治理敘事**上是「小黑屋」（可逆隔離）；**法律**上是 DSA 第三類執法的 visibility restriction。〔排序報告〕三者指的是同一個動作，但分屬三個尚未對齊的世界。

## 7.6 史前史：權重與 MAT（2018）

首頁排序並非從演算法開始。2018 年的第一代機制是**使用者權重**：不同使用者的讚賞有不同倍率，集體評價決定首頁排序（第 02 章）。它因套利風險（形同洗錢）而與 MAT 脫鉤、最終取消，2019 年新站上線改用演算法排序。〔運營史〕值得一提的是，創辦初期「前兩三年首頁完全沒有 recommendation algorithm」，是社群長大、內容彼此看不見之後，才「不得不重新引入演算法」。〔g0v〕——這正是第 01 章「本想逃離平台，最後重新發明平台」的具體一例。

## 小結

首頁排序是「決定誰被看見」這個治理權力最日常、也最隱形的形式。它與 spam 過濾（二元剔除）必須分開理解，才看得懂「文章還在卻沒人看到」的兩種成因。而降權路徑的正當程序缺口——無通知、無申訴、無救濟——正是本計畫想推動討論的政策前沿。下一章，我們轉向治理的另一條路線：不靠站方、也不靠模型，而是靠社群自己。
