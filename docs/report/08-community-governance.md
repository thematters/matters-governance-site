# 08 ─ 群眾協力：守望相助隊、海巡 bot，與治理的失敗

> 不靠站方、也不靠模型，而是靠社群自己——這條路線是 Matters 最有特色、也最坎坷的治理實驗。本章涵蓋當下的兩條群眾路線（人＋機器），以及歷史上一連串「把治理下放社群」的嘗試與失敗。

## 8.1 三條群眾路線

| 路線 | 主體 | 現況 |
|---|---|---|
| 守望相助隊 | 可信市民（人） | 2026 上線運行 |
| 海巡 bot | 自動巡邏（機器） | 鷹架完成、保守 dry-run 中 |
| 訴訟／陪審團 | 全體或建築師（制度） | 2019–2022 嘗試，已收場 |

## 8.2 守望相助隊：自己的困擾自己解

**起源**：站上最讓人困擾的是色情廣告，運營已手動刪除三萬多則，兩年前導入的過濾演算法也不斷被新變種超克。於是「既然讀者與創作者感到困擾，不如**自己的困擾自己解**？」——設計一個讓使用者可以「打掃」（隱藏）垃圾留言的功能。〔守望幕後〕

**兩條完全分離的路徑**（這是設計上的關鍵）：〔排序報告〕

- **檢舉（`submitReport`）**：任何登入者可檢舉，理由含侵權、違法廣告、歧視仇恨、未成年色情等。落入 `report` 表，**不自動通知管理員、不直接下架**；唯一自動效果是同一留言被 3 個以上不同帳號檢舉時自動折疊。
- **打掃（`communityWatchRemoveComment`）**：需 level-2 權限 + `communityWatch` flag，理由限定 `porn_ad`（色情廣告）／`spam_ad`（濫發廣告），留言立即 `banned`，寫入 `community_watch_action` 存證表，原文保留 7 天後刪除，並以「本則貼文已由守望相助隊檢舉」取代、連到公開的 `/records/{uuid}` 頁面。

**有趣的設計順序**：團隊「先給移除權、再加檢舉權」——因為一開始不確定使用者會怎麼操作這個「權力」。〔排序報告〕

**防作惡的兩個機制**：(1) 公開布告欄 `/records/`（預設遮蔽原文，避免二次傳播）；(2) 管理員覆核（upheld／reversed／reason_adjusted）＋ 申訴狀態。〔排序報告〕〔守望幕後〕「透明紀錄的目的不是擴大懲罰，而是讓刪除、申訴與覆核之間形成可查核的社群自治循環。」〔守望幕後〕

**一週實驗數據**（公開後一週）：〔排序報告〕

- 二十餘名使用者報名，打掃 **339 則**違規內容（色情廣告 219、濫用廣告 20）。
- 單一使用者「打掃超人」一人檢舉 **304 則，佔近九成**。
- 被打掃的違規留言中，**70% 由四個機器操縱的帳號產生**。
- **所有被打掃帳號均無人申訴。**

後續累積到「將近六百則」，色情廣告約佔九成，仍是零申訴。〔演算法筆記〕

**v1 的紅線**：AI **不得自動移除**，只能建議候選；站方保留覆核與復原的最終權。〔repo〕

## 8.3 海巡 bot：機器當隊員，但綁上煞車

「海巡 bot」是守望相助隊的自動化延伸，**重用第 03 章的留言模型作為偵測器**（確保「bot 動作的對象」與「模型評分的對象」永不分歧）。〔repo〕

**兩級處置**：
- **Tier-1 檢舉**：分數 ≥ 0.55（正式環境提高到 0.80）→ 送檢舉；3 個檢舉自動折疊。
- **Tier-2 打掃移除**：分數 ≥ 0.80 → 移除；**預設關閉**，需治理簽核才開。
- **Ring 閘門**：覆寫分數，處理集團洗版（第 04 章）。

**護欄（預設全開）**：〔repo〕
- `DRY_RUN`、`KILL_SWITCH`、`ENABLE_REMOVE=false`、預設 staging 環境。
- **反轉率煞車**：若近期守望相助隊的「被推翻率」> 10%，bot 直接罷工——把人類複核的訊號變成自動化的即時安全閥。
- 流量上限（每次最多動作數、候選數）。

**運作**：GitHub Actions 每 6 小時巡邏；三階段 rollout（觀察 → 只檢舉 → 完整），靠 repo 變數推進、不改程式。基線 dry-run（72 則留言）：6 則達 Tier-2（全是真廣告、0 偽陽）、1 則 Tier-1 偽陽。〔repo〕

## 8.4 閉環：群眾標註 → 模型

守望相助隊的公開移除紀錄，**本身就是模型的訓練真相**：被確認移除的＝正樣本，被推翻／復原的＝高權重負樣本（hard negative）。模型回訓後變強、誤殺更少、人類負擔下降——形成一個自我改進的循環。〔排序報告〕〔repo〕主持人借用唐鳳 civic.ai 的比喻，把這個由群眾集體智慧創生的機器隊員稱為「**地神 Kami**」。〔演算法筆記〕

## 8.5 歷史的失敗：把治理下放社群，為什麼難

群眾治理在 Matters 不是新事，而是一連串的嘗試與收場：〔運營working〕〔運營史〕

**社區訴訟制度（2019.08）→ 失敗**：任何人可發訴狀、獲 20 人打賞啟動、由達標使用者投票。問題很快浮現：訴狀難被發現、投票人數過少、**抱團式人情拉票**、**註冊小號灌票**。

**建築師陪審團（2021.03）→ 收場**：改由 21 位社區活躍使用者組成陪審團裁決爭議。但運作不到一年即停，兩大敗因：〔運營working〕
1. **專業度被忽略**：建築師都是普通使用者，不知如何給出有效裁決，多從主觀意見出發，難以收斂；站方無法判斷的模糊案件（「他罵我是狗算不算侮辱」），交給另一群普通人同樣無解。
2. **內部的「江湖」**：建築師之間也站隊、爭吵、形成派系。最尖銳的洞察是——「**站方最花時間的，不是訴訟案本身，而是陪審團內部的爭吵與調停。**」

**沒有第二屆。** 但團隊保留了價值判斷：「讓社群的使用者來決定社群的事情，還是一個很珍貴的價值。只是我們到底要用什麼樣的制度，讓這個價值延續下去。」〔運營working〕

**遊牧者計畫（2023）→ 舞弊風險**：57 提案、1,200+ 使用者投票分配公共基金，但**監測到 5 組重複 IP 灌票**，再次暴露「網路建帳號零成本」對任何集體決策的腐蝕。〔運營working〕

## 8.6 規模悖論

這些群眾治理的設計，有一個共同的天花板：**它們幾乎只在小社群成立。**〔小黑屋〕〔排序報告〕

> 守望相助這種「賦權使用者自助掃除」的設計，放到 Facebook、Threads、Reddit 不可能——「數大必有白癡」，光是審核誰能當隊員就會逼瘋運營；若 Matters 多 100 倍人，這設計可能必須落日。〔小黑屋〕

主持人因此把它連到一個更大的主張：「世界不能只有大平台；許多有意義的多元實驗，發生在邊緣、在地、特別的地方。」〔排序報告〕**賦權的邊界，最終是永續模式的邊界。**

## 8.7 使用者的聲音

深度訪談（5 位資深使用者，匿名）揭露了治理被使用者真實感受的樣子：〔訪談〕【脫敏】

- **檢舉是自衛，不是他律**：常用檢舉的使用者「都檢舉我文章下面的留言」（色情帳號、或看似評論其實是 bot 生成的騷擾），而非熱心糾察別人。
- **共治失能的擔憂**：一位前建築師對陪審團共治的反思最有分量——「這種共治具體達到什麼效果好像不特別明顯，所以我積極性不高」；核心擔憂是信任與透明的崩塌：「在網路上共治，很多人不用真名……但有人是在假扮一個真人，這樣混在一起，治理的結果可能就沒有結果了……如果我有七八個小號讓小號都選我，共治就變成鬧劇。」她也點出缺乏可持續的激勵：「沒有 likecoin 的價值，誰也不會放那麼多心思。」
- **「無審查」是最強的隱性激勵**：「最重要的就是沒有審查的表達！」但緊接著的期待是——「雖然你們沒有審查，但你們依舊保護我們每個人的身份、保護我們每個人的安全。」沒有私訊功能「也非常好，讓我覺得 safe」。【政治敏感】（境外華人身份安全，須嚴格匿名）
- **官方為先、用戶為後**：一位使用者親歷「正義使者」用不堪詞語私訊霸凌的負面經驗，結論是治理「還是得官方為先，用戶為後，就像官方是里長，里民有意見可寫信箱」。
- **LikeCoin 退潮後，留下的人更單純**：「之前為了 like 寫文章的都離開了，現在單純為了社群交流留下來，互動起來比較沒壓力。」

## 小結

群眾協力是 Matters 治理最動人也最艱難的一面：守望相助隊用透明問責把「賦權」做得可控，海巡 bot 用反轉率煞車把自動化綁上人類訊號，而歷史上的訴訟與陪審團則誠實地記錄了「把裁決下放普通人」的失敗。它們共同指向一個結論——**最便宜有效的治理是好的社群設計，但它的天花板是規模與永續**。下一章，我們回到讓這一切有意義的底層：當治理被濫用時，內容還能不能活下來。
