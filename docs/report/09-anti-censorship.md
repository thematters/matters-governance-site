# 09 ─ 抗審查：內容定址、三層韌性與 Hack the Platform

> 「有一些記者都會在上面發文，即使記者不見了，文章可能會比記者活得久一點。」〔TWIGF〕

如果平台可以審查垃圾，它原則上也可以審查任何內容——包括它不該審的。抗審查不是反對治理，而是替「萬一治理被濫用」留一條後路：**就算平台把你下架，內容仍在你手上。** 本章說明 Matters 怎麼用技術把這條後路做出來。

## 9.1 為什麼需要抗審查

中心化平台掌握了內容篩選與審查權，又可輕易竄改或移除內容——劍橋分析事件、土耳其封鎖維基百科是常被引用的例子。在這個結構下，創作者其實不真正擁有自己的內容。〔去中心〕對華語世界而言還有第二重壓制：「在自由世界，重要內容不容易真正觸達讀者；在威權世界，重要內容無法存在。」〔g0v〕

## 9.2 內容定址 vs 位置定址

抗審查的技術核心，是一個定址方式的轉換：〔去中心〕

| | 位置定址（HTTP） | 內容定址（IPFS） |
|---|---|---|
| 內容綁定 | 特定 IP／網域 | 內容指紋（CID） |
| 如何失效 | **刪伺服器檔案、或改寫 DNS** | 只要還有節點持有該指紋，就讀得到 |
| 抗審查性 | 低 | 高 |

> IPFS 把內容經演算法產生唯一的指紋，「只要網絡中存在持有該內容指紋資料的節點，內容便具備抗審查性與續存性，使傳統的網域與 URL 封鎖失效」。〔去中心〕

實務上甚至觀察到，Google 搜尋結果有時把 IPFS 版本的排序置於原始網域之前。〔抗審查演講〕一句白話替「CID」做了最好的科普：「你今天文章被刪掉……在站外你還是可以透過這個**指紋**回去看，但在主站它就會消失。」〔TWIGF〕

## 9.3 三件技術：IPFS、IPNS、ENS

Matters 的去中心發佈分三個階段長成：〔去中心〕

- **IPFS（2018.10）**：每篇文章發布即轉成完整 HTML、上傳 IPFS 節點，內容變成公開且可被其他節點分發；網頁路徑形如 `https://{節點}/ipfs/{CID}`。內容轉換工具已開源為獨立專案 `ipns-site-generator`；節點代管從自管改用 Storcha 與 Pinata（pin＝告訴網絡「這內容別刪、要的話來這找」）。
- **IPNS（2022.09）**：CID 會隨內容更新而改變，作者無法給讀者一個「永久且自動更新」的連結。IPNS 創造一個固定地址，指向「可隨時更新的指紋指針」——像網址一樣，輸入後永遠看到最新內容，且能像 RSS 一樣被訂閱（如用 Planet 閱讀器）。Matters 把每位使用者的內容集中產生靜態 HTML 網站，IPNS 指向它。
- **ENS（2022.12）**：IPNS 指紋過長難記，於是用以太坊的 ENS 名稱指向使用者的 IPNS，例如 `vitalik.eth`，好記又好分享。

## 9.4 三層內容韌性模型

把抗審查放進治理框架，可以看成由外而內的三層：〔排序報告〕

| 層 | 內容 | 治理對象？ |
|---|---|---|
| **Layer 3 可被收編** | matters.town 首頁、搜尋、推薦 | 是——脆弱於法院、ISP、政策（§230／DSA 主要控制點） |
| **Layer 2 不可下架** | 所有文章寫入 IPFS，分散節點錨定 | 否——「只要一個節點存活，內容就在」 |
| **Layer 1 另闢蹊徑** | ENS contenthash、IPFS gateway、Tor onion、Fediverse mirror | 否——站外多重入口 |

由此導出一個對政策極重要的推論：

> 「**憲法言論保障的射程，到得了 Layer 2（內容存在），卻到不了 Layer 3（入口排序）。**」〔排序報告〕

換句話說：抗審查技術保住了「內容不被消滅」，但「沒人看得到」這件事——也就是 Layer 3 的排序與可見度——仍在平台與監管的掌握中。這正好接回第 07 章的正當程序缺口。「內容可以靠 IPFS、Tor 拯救，但『沒人看得到』；而『看不見』的成本，遠低於『被刪除』。」〔排序報告〕

## 9.5 分層治理：本計畫的核心倡議

這個三層結構，導出了計畫最重要的政策主張：**內容層可封禁、基礎設施層仍續存。**〔提案〕

> 「平台可封禁內容，使用者卻仍持有儲存內容之去中心網址，由此分層處理，平衡權益與平台管理。」〔提案〕

也就是說，平台「能管理公共入口與排序」與個人「不被徹底消音」並不矛盾——只要把治理權力限制在 Layer 3，而把內容存續保障在 Layer 2。這是一個可被其他中小型平台複製的設計原則。

## 9.6 Hack the Platform：四個小實驗

在 g0v 場次，團隊把「不把平台當唯一單位、改造可重用元件」的精神，具體化為四個對外開放的小實驗：〔g0v〕

1. **Matters RSS 訂閱**（`rss.matters.town`）：把站內訂閱關係輸出成標準 RSS，讓讀者脫離平台介面追蹤作者。
2. **守望相助隊**（`community-watch.matters.town`）：把治理現場部分開放給市民共看共議（第 08 章）。
3. **洋蔥小站**（Tor onion service 鏡像）：在封鎖環境下仍可低識別地讀寫 Matters。
4. **IPFS 救援轉換器**（`matters-safety-guide.pages.dev`）：把上鏈的 IPFS 文章一鍵轉成可離線、可備份的閱讀格式。

這四個小東西共同示範了一種替代想像：**與其打造「一個大型、漂亮的平台」，不如做「隱形的基礎設施」，讓許多小型社群都能重複使用。**〔抗審查演講〕

## 9.7 蜜罐悖論：抗審查的代價

抗審查不是免費的。它把大量敏感、危險的文本集中到同一處，使平台成為惡意攻擊的高價值目標——這就是第 01 章的**蜜罐悖論**。〔抗審查演講〕

> 「即便架構更安全，光是聚焦的注意力本身就具有危險性。」確實存在案例是在抗審查平台上發布的貼文，後來被用作法庭證據。〔抗審查演講〕【政治敏感】

正因如此，治理（清理垃圾、維持公共性）對一個抗審查平台不是選項，而是生存前提。抗審查與審查，在這裡再一次合流成同一件事的兩面。

## 小結

抗審查的技術底層，是把「內容存續」與「平台入口」分層：IPFS／IPNS／ENS 保住 Layer 2 的內容不被消滅，而治理權力被限制在 Layer 3 的排序與可見度。這個分層既是 Matters 的工程現實，也是本計畫對「打詐與言論自由如何相容」的核心政策回答。下一章，我們看這套理想在 2025 年春節遭遇的真實考驗——當網域本身被封鎖時，會發生什麼。
