# 12 ─ 開源與資料集公開：分層揭露的策略

> 本計畫的方法論本身就是它的主張：把一般平台視為機密的審查演算法攤開來。但「攤開」不是全有全無，而是一道需要精心設計的分層。

## 12.1 開源 ≠ 透明 ≠ 共創

主持人反覆強調一個三段遞進：〔g0v〕

1. **開源 ≠ 透明**：「如果只是把程式碼丟出來，其實不等於透明。因為一般使用者不一定能讀懂程式碼……開源只是第一步，後面還需要文件、案例、可理解的說明、申訴管道。」（這正是本網站存在的理由。）
2. **開放 ≠ 共創**：資深使用者 Denken 從使用者端補上最深刻的一刀——「開源 ≠ 社群參與；開放 API ≠ 使用者能影響治理決策。」他親身參與多次公開工程會議，卻發現「這個開源最終沒有促成社區裡的共創……到最後常常只發現只有我。」〔TWIGF〕

換句話說，透明開盒的真正工作，不在「公開」那一刻，而在公開之後把它變得**可讀、可討論、可參與**——而後者遠比前者難。

## 12.2 為什麼 Matters 敢開源

大部分平台不敢公開審查演算法，因為一開源，詐騙與垃圾帳號就會反過來攻擊、繞過防線。Matters 的理由很反直覺：〔TWIGF〕

> 「我們想說我們已經太小了，開源出去，feedback 已經換到更多了，再多一點點沒關係。」

也就是說，對一個中小型平台，**開源換到的社群信任與外部回饋，已經大於被繞過的邊際風險**。這是大平台算不過來、小平台卻划算的一筆帳。

## 12.3 分層開源：程式碼公開，敏感參數延時

但「敢開源」不等於「全部公開」。折衷方案是**分層開源**：〔排序報告〕

- 程式碼與治理流程可公開、可檢驗。
- **可被直接濫用的敏感參數**（如 `spamThreshold`、Ring 規則、門檻掃描細節）**延遲六到十二個月才釋出**，或抽象化處理。
- `spamThreshold` 被明確視為「敏感的營運資產」。

這道分層導出一句略顯悲涼、卻誠實的結論：

> 「維持抗審查平台，往往需要為了安全而犧牲透明。」〔排序報告〕

本報告全程的【內部】標記，就是這道分層在文件層級的落實。

## 12.4 垃圾內容靜態資料庫公開

開源的下一步是資料公開。`matters-spam-dataset` 是一個規劃中的去識別研究資料集：〔repo〕

- **格式與分發**：靜態批次釋出到 Hugging Face（gated，需同意資料使用協議），版本以 git tag 標記；首版 **v2026.06 約 4 萬筆**（文章為主，留言／動態增量加入）。
- **去識別**：每筆 id 是 HMAC 雜湊（release salt 發布後即銷毀、不可逆）；文本中的聯絡方式正規化為佔位符（`<PHONE>`／`<EMAIL>`／`<LINE_ID>`／`<URL host=…>`…）；時間粗化到月。
- **標籤結構**：二元 `spam`／`ham` + 研究用 `spam_category`（seo／gambling／pharma／scam／porn／mlm…）+ 標籤來源 + 訓練權重（2.0／1.0／0.5）+ 凍結的 test split。
- **誠實揭露雜訊**：資料卡主動寫明早期「作者被限制 → 內容全標 spam」的代理標籤瑕疵，誤掃中文政治／學術／創作長文；約 5,000 筆已修正，未修正者仍可能殘餘 5–15% 誤標。
- **現況**：⚠️**尚未正式發布**——P0（法律審查 ToS 再散布條款與資料使用協議）仍待辦，呈現時不可講成已上線。〔repo〕

這個資料集，正是計畫提案裡「垃圾內容靜態資料庫公開、作為前線打詐最佳實務」的具體載體。〔提案〕〔演算法筆記〕

## 12.5 授權策略

依產出類型分流：〔提案〕

| 產出 | 授權 |
|---|---|
| 程式碼 | MIT 或 AGPL-3.0（視軟體物料清單） |
| 文件（含本報告、網站） | CC BY 4.0 |
| 研究資料 | 依敏感度採開放資料或研究專用授權，並去識別化 |

## 小結

透明開盒是一道光譜，不是開關：程式碼全公開、敏感參數延時、資料去識別後分級釋出，並在公開之後持續把它翻譯成可讀、可討論、可參與的形式。這套分層揭露，既是方法，也是本計畫對「安全與透明如何兼得」的回答。下一章收束全報告。
